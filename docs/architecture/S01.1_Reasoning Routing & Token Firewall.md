
#fcp

1. Objective
    
    Isolate the LLM's internal cognitive monologue from its structural JSON output in real-time, while strictly managing the context window boundary. For reasoning models (e.g., DeepSeek-R1, Qwen3-Reasoning), the Engine leverages native API parameters to route the reasoning trace to the TUI. Crucially, this layer acts as the Token Firewall, preventing context exhaustion via strict generation limits.
    
2. Architecture & Design Rules
    

A. The Reasoning Router (`src/engine/ollama.rs`)

We do not waste CPU cycles parsing `<think>` tags with custom regex. We use Ollama's native separation capabilities.

- **The Payload:** When sending the HTTP POST request to `/api/chat`, the client must inject the `"think": true` flag into the JSON body if `AppConfig.enable_reasoning_fsm` is active.
    
- **The Stream Processor:** The HTTP client must request `stream: true`.
    
    - As chunks arrive, if the chunk contains the `message.thinking` field, that string is instantly pushed to the `stderr` or the TUI `mpsc` channel for live rendering.
        
    - If the chunk contains the `message.content` field, it is appended to the internal `json_payload` buffer.
        
- **The Clean Return:** Upon receiving the `done: true` chunk, the Engine returns _only_ the `json_payload` string back to Layer 2, guaranteeing the Gatekeeper never accidentally tries to parse the monologue as JSON.
    

B. The Fallback Protocol (Standard Models)

If `AppConfig.enable_reasoning_fsm` is false (e.g., using a standard Llama 3.2 model), the `"think"` parameter is omitted from the request. All incoming chunks naturally arrive in `message.content` and are appended directly to the JSON buffer.

C. The Token Firewall (The Soft Limit)

Because we removed the `llama.cpp` C-API, we cannot perfectly count tokens _before_ generation. We must protect the VRAM defensively.

- **Mechanism:** When assembling the payload for Ollama, the client must explicitly set the `options.num_predict` parameter.
    
- **The Calculation:** `num_predict` = `AppConfig.num_ctx` - (An estimated token count of the current `[CHAT_STACK]`).
    
- **The Guillotine:** If Ollama hits the `num_predict` limit, it will return `done_reason: "length"`. If the Orchestrator sees this reason, it immediately yields an `Err(FcpError::ContextExhaustion)`. The Fuckup Loop intercepts this, triggers a synchronous `Reflect` compaction task (Block S02), and retries the generation.
    

3. Acceptance Criteria
    

- [ ] When `enable_reasoning_fsm` is true, the `OllamaClient` successfully streams the `message.thinking` payload to a separate `mpsc` channel while buffering `message.content` into the final JSON string.
    
- [ ] When `enable_reasoning_fsm` is false, the Engine bypasses the thinking logic entirely, collecting standard JSON output without crashing.
    
- [ ] The `OllamaClient` accurately sets the `num_predict` parameter in the API options payload to prevent context overflow.
    
- [ ] Receiving a `done_reason: "length"` from the Ollama API successfully aborts the current tool execution and yields an `FcpError::ContextExhaustion` to trigger the compaction loop.
