
#fcp

1. Objective
    
    Implement a high-speed, concurrent, in-memory cache to stage facts extracted during the Reflect state. This acts as an "Attention Filter." Facts live here for a strictly defined Time-To-Live (TTL). If the LLM does not explicitly promote them to the permanent vault before the TTL expires, they evaporate. To survive terminal reboots, the cache must serialize its unexpired state to a workspace-isolated disk file on a configurable interval, using absolute epoch time to cleanly discard stale data on boot.
    
2. Architecture & Design Rules
    

A. The Cache Engine & Time Physics (`src/memory/ephemeral.rs`)

We use `moka`, an async-aware, concurrent cache. Because `moka` uses relative monotonic time, we must wrap our values in absolute `SystemTime` to survive process restarts.

Rust

```
use moka::future::Cache;
use serde::{Serialize, Deserialize};
use std::time::{Duration, SystemTime};
use std::sync::Arc;
use crate::config::AppConfig;

#[derive(Clone, Serialize, Deserialize)]
pub struct EphemeralEntry {
    pub content: String,
    pub inserted_at: SystemTime,
}

#[derive(Clone)]
pub struct EphemeralStore {
    // Key: String (e.g., "ephemeral:{timestamp}.{tag}")
    cache: Cache<String, EphemeralEntry>,
    config: Arc<AppConfig>,
}

impl EphemeralStore {
    pub fn new(config: Arc<AppConfig>) -> Self {
        let cache = Cache::builder()
            // Hard RAM boundary: Maximum 10,000 active entries
            .max_capacity(10_000)
            // moka's internal relative TTL (evicts from RAM while process runs)
            .time_to_live(Duration::from_secs(config.ephemeral_ttl_secs))
            .build();
            
        Self { cache, config }
    }
}
```

B. The Key Taxonomy (Routing Strings)

The cache is a flat key-value store. We enforce structure through strict key naming conventions, allowing the Orchestrator to scan for specific data types.

- `summary:running`: The active, compressed summary of the current chat stack. Overwritten every Condensation cycle. Never expires while the session is active.
    
- `ephemeral:{unix_timestamp}.{tag}`: A staged fact. Example: `ephemeral:1711440000.user_preference`.
    
- `sensory:{unix_timestamp}.{source}`: Temporary observations, like a newly dropped file name that hasn't been fully read yet.
    

C. Disk Snapshots (The `bincode` Serialization)

The dreadnought must be able to shut down without losing its active short-term memory. The target file is strictly `.fcp/ephemeral_{workspace}.bin`.

- **Write (On Interval or SIGINT):** A background `tokio` task selects over a `tokio::time::interval` and the `CancellationToken`. When triggered, it iterates over the cache. It filters out natively expired keys, serializes the remaining `(String, EphemeralEntry)` tuples using `bincode`, and performs a File Truncate write. It never appends.
    
- **Read (On Boot):** The Orchestrator deserializes the `.bin` file. It calculates `SystemTime::now().duration_since(entry.inserted_at)`. If the difference is greater than `config.ephemeral_ttl_secs`, the fact is legally dead and dropped. Survivors are injected into the fresh `moka` cache.
    

D. Tooling Integration (`NOUN:VERB` Syntax)

The Gatekeeper exposes this cache to the LLM via two specific tools:

- `memory:stage(content, tag)`: Inserts a new string into `moka` using the `ephemeral:{timestamp}.{tag}` format, capturing the current `SystemTime`.
    
- `memory:commit(tag)`: Reads all keys matching the tag, formats them, and returns them to the Orchestrator to be appended to the permanent Obsidian Vault (Tier 3). It instantly `.invalidate()`s those keys from `moka` so they aren't written twice.
    

E. Shutdown Hook

The background `bincode` task must explicitly prioritize the `CancellationToken`. If the token triggers, it performs one final synchronous flush to disk and gracefully terminates.

3. Acceptance Criteria
    

- [ ] Executing the `memory:stage` tool successfully stages a fact in `moka` wrapped in an `EphemeralEntry` containing the absolute Unix timestamp.
    
- [ ] Facts placed in the cache automatically become irretrievable from RAM exactly after `config.ephemeral_ttl_secs` has elapsed.
    
- [ ] The background Tokio task successfully fires every `snapshot_interval_secs`, generating the `.fcp/ephemeral_{workspace}.bin` file containing only unexpired entries.
    
- [ ] Booting the binary after an artificial OS time-shift greater than `ephemeral_ttl_secs` successfully ignores the expired `.bin` records, booting with an empty `moka` cache.
    
- [ ] Triggering the global `CancellationToken` from the main thread successfully forces the background task to perform an immediate flush to disk before terminating.
