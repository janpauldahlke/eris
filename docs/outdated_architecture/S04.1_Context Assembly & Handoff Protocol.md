
#fcp

1. Objective
    
    Define the exact string topology of the system prompt and the rigid JSON contract that binds Layer 1 (The LLM) to Layer 2 (The Subconscious). If the context assembly is bloated, the model loses focus. If the handoff protocol is loose, the loop breaks. This block engineers the injection payload and the execution router.
    
2. Architecture & Design Rules
    

A. The Context Topology (String Assembly)

The Orchestrator dynamically compiles the system prompt every turn. It is a modular template injected with live memory.

- `[DIRECTIVE]`: Absolute core rules (e.g., "Output ONLY valid JSON.", "Do not apologize."). Sourced by reading `00_Core/Identity.md` directly from disk per turn (enabling hot-reloading without binary restarts).
    
- `[ACTIVE_STATE]`: Defines the current cognitive mode based on the `AgentState` enum (e.g., "You are in REFLECT mode. Analyze the stack and extract key facts.").
    
- `[THE_LENS]`: High-signal, semantic matches retrieved from Qdrant (Tier 3). Injected only if a `memory:query` tool was recently executed.
    
- `[EPHEMERAL_WORKSPACE]`: Active facts staged in the `moka` cache (Tier 2). Keeps the LLM aware of what it has recently learned but hasn't committed yet.
    
- `[CHAT_STACK]`: The active conversation. Begins with `session:running_summary` (if condensation occurred in Block S02), followed by the latest User and Assistant turns.
    

B. The LLM Output Schema (The JSON Contract)

Because the Reasoning Router (Block S01.1) natively separates the cognitive monologue via the Ollama API, the Orchestrator expects a mathematically clean JSON object to emerge from the `EngineResponse.content`. The LLM is strictly prompted to output only this schema:

JSON

```
{
  "status": "CONTINUE_TASK | WAIT_FOR_USER | INITIATE_REFLECTION",
  "message_to_user": "String (Optional. Null if calling a silent tool)",
  "tool_calls": [
    {
      "name": "memory:query",
      "args": { "query": "rust ownership", "filter_tag": "procedural" }
    }
  ]
}
```

C. The Handoff Router (Rust Logic)

When the Orchestrator receives the parsed JSON, it evaluates the status flag to determine the next loop iteration.

Rust

```
pub enum LoopDirective {
    ExecuteTools(Vec<ToolCall>),
    HaltAndAwaitInput(Option<String>),
    ShiftToReflection,
    RecoverFromFuckup(String),
}

impl Orchestrator {
    pub async fn process_llm_response(&mut self, response: LlmResponse) -> LoopDirective {
        match response.status.as_str() {
            "CONTINUE_TASK" => {
                if response.tool_calls.is_empty() {
                    LoopDirective::RecoverFromFuckup("FUCKUP: CONTINUE_TASK requires tool_calls.".into())
                } else {
                    LoopDirective::ExecuteTools(response.tool_calls)
                }
            },
            "WAIT_FOR_USER" => LoopDirective::HaltAndAwaitInput(response.message_to_user),
            "INITIATE_REFLECTION" => {
                self.set_state(AgentState::Reflect);
                LoopDirective::ShiftToReflection
            },
            _ => LoopDirective::RecoverFromFuckup("FUCKUP: Invalid status flag.".into())
        }
    }
}
```

D. Context Interception (The Fuckup Injector & Breaker Switch)

If the router yields `RecoverFromFuckup`, or if a tool execution yields an `Err()`:

1. **The Orchestrator:** Increments its internal `recovery_count`.
    
2. **The Breaker:** If `recovery_count > AppConfig.max_recovery_attempts`, it forces a `WAIT_FOR_USER` bailout to prevent infinite looping and context exhaustion.
    
3. **The Injector:** If under the limit, it creates a synthetic "User" message containing the exact error string (`FUCKUP DETECTED: [stderr trace]`), pushes it to the `[CHAT_STACK]`, and immediately fires the LLM again without waiting for real `stdin`.
    
4. Acceptance Criteria
    

- [ ] The prompt builder successfully concatenates the `[DIRECTIVE]`, `[ACTIVE_STATE]`, `[EPHEMERAL_WORKSPACE]`, and `[CHAT_STACK]` without exceeding token limits.
    
- [ ] When the LLM outputs `WAIT_FOR_USER`, the Orchestrator successfully halts inference and prints `message_to_user` to `stdout`.
    
- [ ] When the LLM outputs `CONTINUE_TASK` with a valid tool, the Orchestrator executes the tool, appends the result to the context, and automatically re-triggers inference.
    
- [ ] Emitting an invalid status flag successfully catches the error in Rust, shifts to the recovery loop, and forces the LLM to correct its JSON schema on the next turn.
    
- [ ] Modifying `00_Core/Identity.md` during an active `WAIT_FOR_USER` state successfully alters the `[DIRECTIVE]` string on the very next turn without restarting the binary.
