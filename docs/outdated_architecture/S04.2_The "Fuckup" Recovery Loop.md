
#fcp

1. Objective
    
    Engineer a deterministic, self-healing recursion loop exclusively for Cognitive Faults. When the LLM hallucinates a tool argument or violates the JSON contract, the Orchestrator intercepts the error, shifts into `AgentState::Recover`, and forces the model to read the trace and self-correct. For System Fatalities (e.g., database crashes), the loop immediately aborts, dumps telemetry to `stderr`, and yields control to the user.
    
2. Architecture & Design Rules
    

A. The Error Bifurcation (The Intercept Matrix)

The Subconscious Orchestrator wraps the execution layer in a `match` statement that strictly evaluates the `FcpError` variant.

- **Recoverable (Cognitive Faults):** * `ParseFault`: LLM outputs invalid JSON or trailing commas.
    
    - `SchemaViolation`: JSON is valid, but missing required arguments (e.g., called `memory:query` but forgot the `query` field).
        
    - `ToolFault`: Tool runs but fails due to bad logic (e.g., trying to read a file that the LLM hallucinated, or attempting to write to the `00_Core/` immutable zone).
        
    - _Action:_ Shift to `AgentState::Recover` and increment `recovery_attempts`.
        
- **Fatal (System Fatalities):**
    
    - `Io`, `Config`, `EngineFault`, `NetworkFault`, `VectorDbOffline`, `EmbeddingFault`.
        
    - _Action:_ DO NOT enter the recovery loop. Immediately dump the error trace to `stderr` via `tracing::error!`, shift state to `Idle`, and force a `WAIT_FOR_USER` signal to hand terminal control back to the user. Let the system bleed out visibly.
        

B. The State Transition & Prompt Injection

When a Cognitive Fault is caught:

1. The Orchestrator increments the `recovery_attempts` counter.
    
2. State shifts to `AgentState::Recover`.
    
3. The active context window is not wiped. Instead, the Orchestrator appends a synthetic system message directly to the `[CHAT_STACK]`:
    

Plaintext

```
[SYSTEM OVERRIDE: FUCKUP DETECTED]
Your previous tool call failed. Analyze the stack trace and issue a corrected JSON payload.

ERROR TRACE:
{error_string_from_rust}
```

C. The Bailout Condition (Infinite Loop Protection)

Models can get stuck in a panic spiral, repeatedly trying the exact same broken tool call. We must sever the loop.

- **Mechanism:** Enforce `AppConfig.max_recovery_attempts` (e.g., 3).
    
- **Execution:** If `recovery_attempts >= max_recovery_attempts`, the Orchestrator hard-aborts the cognitive recovery.
    
- **Fallback:** It shifts the state back to `Chat`, dumps the loop trace to the UI, and sets the internal status to `WAIT_FOR_USER`. The agent essentially says: "I tried fixing this 3 times and failed. Need human override."
    
- **Reset:** Upon any successful execution of a tool or a valid turn, the `recovery_attempts` counter MUST be reset to `0`.
    

3. Acceptance Criteria
    

- [ ] Simulating a malformed JSON output (`ParseFault`) successfully routes to the Recovery Loop and injects the parse error into the next prompt without crashing the Rust binary.
    
- [ ] The LLM successfully uses its native `<think>` capabilities (via the Reasoning Router in S01.1) to analyze the injected error string and outputs a corrected JSON payload on the subsequent turn.
    
- [ ] Intentionally requesting a non-existent file 3 times in a row correctly triggers the `max_recovery_attempts` bailout, yielding terminal control back to `stdin`.
    
- [ ] Simulating an `FcpError::VectorDbOffline` immediately bypasses the recovery loop, emits a `tracing::error!` to `stderr`, and halts inference to await user input.
    
- [ ] Upon a successful recovery (a valid tool execution on attempt 2), the `recovery_attempts` counter resets to `0` and the state shifts back to `Chat`.
